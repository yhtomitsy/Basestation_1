Lattice 4:1 MIPI CSI-2 Bridge Soft IP

IP Core Generation


Configuring IP Core in Clarity Designer

  1. Input valid values in the required fields in the Configuration tab.
    • Number of RX Channels : [Read-Only] This is fixed to 4 RX channels.
    • Number of RX Lanes : [User-Input] Specify the number of RX lanes. single, 2- or 4- lane configuration is supported.
    • RX Gear : [Read-Only] Only RX gear 8 is supported in this IP.
    • RX Interface : [Read-Only] This IP only supports CSI2 interface.
    • RX D-PHY Clock Mode : [User-Input] Choose "Continuous" if the input (RX Channel 0) DPHY clock is continuously running in high speed mode. The generated rx dphy byte clock will be used to clock internal logic. Otherwise, choose "Non-Continuous" and a refclk port will be available for connection to an external continuously running clock. The refclk frequency requirement is indicated by the Reference Clock Frequency field.
    • Lane Aligner FIFO Type : [User-Input] Selects the implementation type of the FIFO within the Lane Aligner module. Select EBR if more LUTs are needed for other logic.
    • Enable Lane Aligner : [User-Input] The optional lane aligner module may be enabled if there are large skews among the data lanes (if skew is greater than 1 byteclock). Otherwise, this can be left disabled.
    • Number of TX Channels : [Read-Only] This is fixed to only 1 TX channel.
    • Number of TX Lanes : [Read-only] The number of TX lanes is the same as the "Number of RX Lanes" indicated.
    • TX Interface : [Read-Only] Only supports CSI2 interface.
    • TX Gear : [User-Input] Select gear 16 for higher data rates; Select gear 8 for lower logic resources (can only support up to 1.2Gbps per TX data lane).
    • TX DPHY Clock Mode : [User-Input] This configures the TX DPHY Clock Lane behavior. The clock lane is always in high-speed and does not transition to LP state in "Continuous" mode.
    • RX Line Rate : [Read-Only] Automatically computed based on the desired TX line rate.
    • TX Line Rate : [User-Input] Specify the target TX line rate. All other clocks are computed based on this input.
    • DPHY Clock Frequency : [Read-Only] Automatically computed based on the desired TX line rate.
    • TX Byte Clock Frequency : [Read-Only] Automatically computed on from the desired TX line rate.
    • Reference Clock Frequency : [Read-Only] Automatically computed based on the desired TX line rate.

  2. Input valid values in the required fields in the Video tab.
    • Merging Option : [User-Input] Choose "Left-Right" to get side-by-side images from both channels. Choose "Virtual" to interleave packets from the two rx channels. In virtual interleave, the channel ID of the packets will be replaced by the value specified in the Virtual Channel ID fields.
    • Virtual Channel ID (ch0) : [User-Input] Specify the Virtual Channel ID for channel 0.
    • Virtual Channel ID (ch1) : [User-Input] Specify the Virtual Channel ID for channel 1.

  3. DPHY timing may be configured in the Protocol Timing Parameter tab. The values indicated are in terms of number of byteclock cycles.
    • t_HS-PREPARE : [User-Input] This is the time that the transmitter drives the Data Lane LP-00 Line state immediately before the HS-0 Line state starting the HS transmission.
    • t_HS-ZERO : [User-Input] This is the time that the transmitter drives the HS-0 state prior to transmitting the Sync sequence.
    • t_CLK-PRE : [User-Input] This is the time that the transmitter drives the HS clock prior to any associated Data Lane beginning the transition from LP to HS mode.
    • t_CLK-POST : [User-Input] This is the time that the transmitter continues to send HS clock after the last associated Data Lane has transitioned to LP Mode. Interval is defined as the period from the end of THS-TRAIL to the beginning of TCLK-TRAIL.
    • tINIT_SLAVE Value : [User-Input] Specify the wait time period needed for DPHY slave initialization. No packets will be passed across the bridge during this time.
    • Bypass tINIT_SLAVE counter : [User-Input] this counter may be disabled, if the initialization period of the DPHY slave is not required. The counter starts once the TX PLL has locked.

  4. Internal signals may be ported out as aid in debugging by selecting the corresponding checkboxes in the Miscellaneous Ports tab.
    • Enable miscellaneous status signals : [User-Input] Click to enable the status signals.
    • TX PLL lock : This signal indicates the status of the TX PLL lock state.
    • Initialization Done : This signal asserts once the tINIT slave counter has reached the number of byteclock cycles indicated in the tINIT_SLAVE Value field. No DPHY packets are passed across the bridge until this signal asserts.
    • MUX0 short packet enable : This signal is a pulse that indicates successful reception of a short packet from RX channel 0 (Rx Channel 2, if mux_sel_i is asserted).
    • MUX0 long packet enable : This signal is a pulse that indicates successful reception of a long packet from RX channel 0 (Rx Channel 2, if mux_sel_i is asserted).
    • MUX0 HS sync pattern detected : This signal indicates successful detection of the DPHY synchronization pattern (0xB8) from Rx Channel 0 (Rx Channel 2, if mux_sel_i is asserted) data lanes.
    • MUX1 short packet enable : This signal is a pulse that indicates successful reception of a short packet from RX channel 1 (Rx Channel 3, if mux_sel_i is asserted).
    • MUX1 long packet enable : This signal is a pulse that indicates successful reception of a long packet from RX channel 1 (Rx Channel 3, if mux_sel_i is asserted).
    • MUX1 HS sync pattern detected : This signal indicates successful detection of the DPHY synchronization pattern (0xB8) from Rx Channel 1 (Rx Channel 3, if mux_sel_i is asserted) data lanes.
    • Byte data valid : IF this signal is asserted, this indicates that data is currently being read from the line buffers and being sent out to the tx channel for transmission.
 


Generating IP Core in Clarity Designer

  1. After configuring the IP Core, Click Configure to apply the parameters to the IP.
  2. Click Close. The IP will be generated.
 

Implementing the IP Module Using Lattice Diamond


Instantiating the Core

The generated CSI2 to CSI2 IP core package contains Verilog files. The Verilog HDL files include black-box (<username>_bb.v) and instance (<username>_inst.v) templates that can be used to instantiate the core in a top-level design.

An example RTL top-level reference source file that can be used as an instantiation template for the IP core is provided in <project_dir>\csi2_to_csi2_qm_eval\<username>\src\rtl\top\<family>. Users may also use this top-level reference as the starting template for the top-level for their complete design.

A Verilog instance template <instance_name>_inst.v or VHDL instance template <instance_name>_inst.vhd is also provided as a guide for users who want to instantiate the generated soft IP top-level in their own top-level module.


Hardware Evaluation

Lattice's IP hardware evaluation capability makes it possible to create IP cores that operate in hardware for a limited period of time (approximately four hours) without requiring the purchase on an IP license. The hardware evaluation capability is enabled by default. It can be disabled by Project->Active Strategy->Translate Design Settings. The setting is called "Hardware Evaluation" and the options are "Enable" or "Disable".
 
When the Hardware Evaluation feature is enabled in the design, it will generate a programming file that may be downloaded into the device. After initialization, the IP core will be operational for approximately four hours. After four hours, the device will stop working and it will be necessary to reprogram the device to re-enable operation. This hardware evaluation capability is only enabled if the core has not been licensed. During implementation, a license check is performed. If the hardware evaluation feature is disabled, a pop-up window will be displayed indicating a license failure. Click"OK" in the window and the bitstream will not be generated. If a license is detected, no pop-up window is displayed and core generation is completed with no restrictions.


Implementing the Core in a Top-Level Design


As described previously, the top-level file <username>_top.v(vhd) provided in <project_dir>\csi2_to_csi2_qm_eval\<username>\src\rtl\top\<family> supports the ability to implement just the CSI2 to CSI2 IP Block.

Push-button implementation of this top-level is supported via the Diamond project file <username>_top.ldf located in <project_dir>\csi2_to_csi2_qm_eval\<username>\impl\<synplify|LSE>.

This design is intended to provide an indication of the device utilization associated with the core itself and is usually replaced with the user's top-level design.

 

To use the project file:

  • Select Open -> Project under the File tab in Diamond.
  • Browse to <project_dir>\csi2_to_csi2_qm_eval\<username>\impl\<synplify|LSE>in the Open Project dialog box.
  • Select and open <username>. ldf. At this point, all of the files needed to support top-level synthesis and implementation will be imported to the project.
  • One lpf file is available in the \impl\<synplify|LSE> directory <username>_top.lpf.
  • Implement the complete design via the standard Diamond GUI flow, keeping all other options at their default values.

Running Functional Simulation

The functional simulation includes a configuration-specific behavioral model of the CSI2 to CSI2 IP, which is instantiated in an FPGA top level along with some other logic (like PLLs and registers with Read/Write Interface). This FPGA top is instantiated in an evaluation testbench that provides appropriate stimulus for the CSI2 to CSI2 IP core. The testbench files can be found in <project_dir>\csi2_to_csi2_qm_eval\testbench.

 

The generated IP core package includes the configuration-specific behavior model (<username>_beh.v,provided in <project_dir>\csi2_to_csi2_qm_eval\<username>\src\beh_rtl\<family>) for functional simulation.Models for simulation are provided in the corresponding \models folder if required.

 

To run simulation using Active HDL:

  1. Modify the "do" file located in \<project_dir>\csi2_to_csi2_qm_eval\<instance_name>\sim\aldec\
    1. Specify working directory (sim_working_folder).
       i.e. set sim_working_folder "C:/my_design"
    2. Specify workspace name that will be created in working directory.
       i.e. set workspace_name "design_space"
    3. Specify design name.
       i.e. set design_name "DesignA"
    4. Specify design path where the IP Core generated using Clarity Designer is located.
       i.e. set design_path "C:/my_designs/DesignA"
    5. Specify design instance name (same as the instance name specified in Clarity Designer).
       i.e. set design_inst "DesignA_inst"
    6. Specify Lattice Diamond Primitive path (diamond_dir) to where it is installed.
       i.e. set diamond_dir "C:/lscc/diamond/3.8_x64"
    7. Update testbench parameters to customize data size, clock and/or other settings. See additional info related to Testbench Compiler Directives below.
  2. Under the Tools Menu, select Active-HDL.
  3. In Active-HDL window, under the Tools tab, select Execute Macro.
  4. Select the "do" file <\project_dir>\csi2_to_csi2_qm_eval\<instance_name>\sim\aldec\*.do
  5. Select OK.
  6. Wait for simulation to finish.
  7. Below are the testbench directives which can be modified by setting the define in the vlog command in the "do" file.

    Example:
    vlog \
    +define+NUM_FRAMES=60 \
    +define+NUM_LINES=1080 \
    ....
    * NUM_PIXELS : Number of pixels per line
    * NUM_LINES : Number of lines per frame
    * NUN_FRAMES : Number of frames to be transmitted
    *These parameters can be manually included in a "do" file or TB define file.

    Optional:
    The following timing parameters (in ps) are set to minimum based from MIPI specs:
    * DPHY_LPX
    * DPHY_CLK_PREPARE
    * DPHY_CLK_ZERO
    * DPHY_CLK_TRAIL
    * DPHY_CLK_PRE
    * DPHY_CLK_POST
    * DPHY_HS_PREPARE
    * DPHY_HS_ZERO
    * DPHY_HS_TRAIL

    The following timing parameters are related to start of dphy model driving:
    * DPHY_INIT_DRIVE_DELAY : delay from reset deassertion or tinit_done assertion before ch0 starts driving data.
    * CH0_DELAY : number of byteclocks before RX Channel 0 starts transmitting data. Default is 0.
    * CH1_DELAY : number of byteclocks before RX Channel 1 starts transmitting data. Default is 0.
    * CH2_DELAY : number of byteclocks before RX Channel 2 starts transmitting data. Default is 0.
    * CH3_DELAY : number of byteclocks before RX Channel 3 starts transmitting data. Default is 0.
    Data types
    * DT_RAW8 : used to set the data transaction to RAW8
    * DT_RAW10 : used to set the data transaction to DT_RAW10
    * DT_RAW12 : used to set the data transaction to DT_RAW12
    * DT_RGB888 : used to set the data transaction to DT_RGB888
    * DT_YUV420_10 : used to set the data transaction to DT_YUV420_10
    * DT_YUV420_8 : used to set the data transaction to DT_YUV420_8
    * DT_YUV422_10 : used to set the data transaction to DT_YUV422_10
    * DT_YUV422_8 : used to set the data transaction to DT_YUV422_8

    * Default is RAW10 if data type is not specified by user.

    Other parameters
    * PIX_CLK : clock period in ps; used as reference clock
    * DPHY_CLK_PERIOD : used to set the DPHY clock frequency of the design
    * DPHY_LPS_GAP : used to inject LPS delay (in ps) in between HS transactions
    * DPHY_FRAME_GAP : used to inject LPS delay (in ps) in between frames
    * VC_CH0 : used to set virtual channel of DPHY CSI2 model 0
    * VC_CH1 : used to set virtual channel of DPHY CSI2 model 1
    * VC_CH0 : used to set virtual channel of DPHY CSI2 model 2
    * VC_CH1 : used to set virtual channel of DPHY CSI2 model 3
    * LS_LE_EN : used to enable DPHY model transmission of LS/LE short packets

    * User can override the default timing parameters using defines above.

     Refer to UserGuide document for more information.

 


Hardware Evaluation

The Lattice 4:1 MIPI CSI-2 Bridge Soft IP supports Lattice's IP hardware evaluation capability, which makes it possible to create versions of the IP that operate in hardware for a limited period of time (approximately four hours) without requiring the request of an IP license. It may also be used to evaluate the IP in hardware in user-defined designs.


Choose Project > Active Strategy > Translate Design Settings. The hardware evaluation capability may be enabled/disabled in the Strategy dialog box. It is enabled by default.
